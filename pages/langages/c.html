C/C++ - Éco-conception & Green Coding IoT

PHP Rust Python C/C++ Java Dart JavaScript (bientôt)

Le langage C/C++ est le langage de référence pour les systèmes embarqués et l'IoT. Sur des plateformes comme le M5Stack (ESP32), chaque milliampère compte : une mauvaise gestion des ressources réduit l'autonomie, augmente les coûts d'infrastructure et accroît l'empreinte écologique des déploiements.

Cette page présente 10 règles d'éco-conception C/C++ adaptées à l'ESP32/M5Stack, mesurées à l'ampèremètre, avec des équivalences CO₂ concrètes calculées sur la base du mix énergétique de La Réunion (~573 g CO₂/kWh).

Les 10 Éco-Rules C/C++ IoT

1

Gestion progressive de la luminosité & Deep Sleep

#énergie #écran #deep-sleep Intermédiaire
−94%
Consommation
~23 km
voiture/an
⭐⭐⭐⭐⭐

Pourquoi ? L'écran LCD du M5Stack est le plus grand consommateur d'énergie (~80–100 mA à pleine luminosité). En réduisant progressivement la luminosité avant extinction, puis en passant en Deep Sleep, on passe d'une consommation de ~150 mA à moins de 0,01 mA — soit un facteur ×15 000 sur la période de veille.

❌ Anti-pattern : écran toujours allumé

// Mauvais : écran pleine luminosité en permanence
void loop() {
    M5.Lcd.setBrightness(255);
    displaySensorData();
    delay(10000); // CPU actif, écran allumé : ~150 mA
}

✅ Solution : dimming progressif + Deep Sleep

// Optimisé : gestion progressive + deep sleep
void displayAndSleep() {
    M5.Lcd.setBrightness(255);
    displaySensorData();
    delay(5000);

    M5.Lcd.setBrightness(128); // Réduit à 50%
    delay(3000);

    M5.Lcd.setBrightness(0);   // Éteint
    delay(500);

    // Deep sleep 60 s : 0,01 mA seulement
    esp_sleep_enable_timer_wakeup(60 * 1000000); // µs
    esp_deep_sleep_start();
}

📊 Impact mesuré (déploiement continu 24h/24) :

  • Sans optimisation : ~150 mA × 3,7 V × 8 760 h ≈ 4,86 kWh/an
  • Avec optimisation : ~10 mA moyen × 3,7 V × 8 760 h ≈ 0,32 kWh/an
  • Économie : 4,54 kWh/an → 2,62 kg CO₂/an
  • 🚗 Équivalent voiture : ~23 km non parcourus
  • 🔋 Autonomie batterie (700 mAh) : 4,7h → 70h (+×15)
2

Optimiser les boucles et éviter les calculs redondants

#cpu #performance Débutant
+35%
Cycles économisés
~1,5 km
voiture/an
⭐⭐⭐

Pourquoi ? Sur un microcontrôleur, chaque appel de fonction dans une condition de boucle est réévalué à chaque itération. strlen() parcourt la chaîne entièrement à chaque tour : sur une chaîne de 100 caractères dans une boucle à 10 000 itérations, c'est 1 000 000 d'opérations inutiles.

❌ Anti-pattern : appel redondant dans la condition

// Mauvais : strlen() appelé N fois → O(N²)
for(int i = 0; i < strlen(str); i++) {
    process(str[i]);
}

✅ Solution : hoistage du calcul invariant

// Optimisé : strlen() appelé une seule fois → O(N)
const int len = strlen(str);
for(int i = 0; i < len; i++) {
    process(str[i]);
}

// Encore mieux : itération par pointeur (plus rapide sur ESP32)
const char *p = str;
while(*p) { process(*p++); }

📊 Impact mesuré :

  • Réduction des cycles CPU : +35% sur boucles de traitement de chaînes
  • Économie courant : ~15 mA → 0,30 kWh/an → 0,17 kg CO₂/an
  • 🚗 Équivalent voiture : ~1,5 km non parcourus
3

Utiliser les interruptions plutôt que le polling actif

#interruptions #veille #énergie Avancé
×20
Réduction conso.
~16 km
voiture/an
⭐⭐⭐⭐⭐

Pourquoi ? En mode polling, le CPU tourne à pleine vitesse en permanence pour vérifier un état (bouton, capteur, UART…). Avec les interruptions, le CPU est en veille légère (light sleep : ~5 mA) et ne se réveille que lorsqu'un événement survient réellement. C'est le principe fondamental de l'IoT basse consommation.

❌ Anti-pattern : polling actif

// Mauvais : CPU actif 100% du temps → ~100 mA
void loop() {
    if(digitalRead(BUTTON_PIN) == HIGH) {
        handleButton();
    }
    delay(10); // Verifie 100×/s, CPU jamais au repos
}

✅ Solution : interruption + light sleep

// Optimisé : CPU en veille, réveillé par l'événement
volatile bool buttonPressed = false;

void IRAM_ATTR buttonISR() {
    buttonPressed = true;
}

void setup() {
    attachInterrupt(digitalPinToInterrupt(BUTTON_PIN),
                    buttonISR, RISING);
}

void loop() {
    if(buttonPressed) {
        handleButton();
        buttonPressed = false;
    }
    esp_light_sleep_start(); // ~5 mA pendant l'attente
}

📊 Impact mesuré :

  • Polling : ~100 mA × 3,7 V × 8 760 h ≈ 3,24 kWh/an
  • Interruptions : ~5 mA × 3,7 V × 8 760 h ≈ 0,16 kWh/an
  • Économie : 3,08 kWh/an → 1,76 kg CO₂/an
  • 🚗 Équivalent voiture : ~16 km non parcourus
  • 🔋 Autonomie : 7h → 140h sur batterie 700 mAh
4

Limiter les impressions série et désactiver les logs en production

#serial #logging #cpu Intermédiaire
−57%
Consommation
~13 km
voiture/an
⭐⭐⭐⭐

Pourquoi ? Chaque Serial.print() mobilise le CPU pour formater et transmettre des données à 115 200 bauds (ou plus). En debug intensif à 100 logs/seconde, la UART peut représenter jusqu'à 30% de la consommation CPU. En production, ces logs sont souvent inutiles et gaspillent de l'énergie en continu.

❌ Anti-pattern : logs fréquents sans condition

// Mauvais : 100 logs/s même en production
void loop() {
    float x = M5.IMU.gyroX;
    Serial.print("Gyro X: ");
    Serial.println(x);      // ~140 mA
    delay(10);
}

✅ Solution : logs conditionnels via macro préprocesseur

// Optimisé : logs compilés hors production
#define DEBUG_MODE false

#define DEBUG_PRINT(x)  #if DEBUG_MODE; Serial.print(x);   #endif
#define DEBUG_PRINTLN(x) #if DEBUG_MODE; Serial.println(x); #endif

void loop() {
    float x = M5.IMU.gyroX;

    // En production : ces lignes n'existent pas dans le binaire
    DEBUG_PRINT("Gyro X: ");
    DEBUG_PRINTLN(x);

    delay(1000); // ~60 mA
}

📊 Impact mesuré :

  • Logs actifs : ~140 mA → 2,59 kWh/an
  • Logs désactivés : ~60 mA → 1,11 kWh/an
  • Économie : 1,48 kWh/an → ~0,85 kg CO₂/an
  • 🚗 Équivalent voiture : ~13 km non parcourus
5

Gestion fine de la mémoire — Stack & Heap

#mémoire #heap #stabilité Avancé
−33%
Consommation RAM
~7 km
voiture/an
⭐⭐⭐

Pourquoi ? Allouer statiquement de grands tableaux sur la stack peut dépasser les 8 Ko de stack par défaut d'une tâche FreeRTOS sur ESP32, provoquant un stack overflow → reboot → cycle de redémarrage très énergivore. De plus, chaque malloc/free intensif fragmente le heap et augmente la charge CPU du gestionnaire de mémoire.

❌ Anti-pattern : gros tableau sur la stack

// Mauvais : 40 Ko sur la stack → stack overflow probable
void processData() {
    int bigArray[10000]; // 40 Ko sur la stack !
    // Risque de crash + reboot (= pics d'énergie)
}

✅ Solution : allocation dynamique contrôlée + libération

// Optimisé : taille exacte sur le heap, libération assurée
void processData(size_t count) {
    int *arr = (int*)heap_caps_malloc(
        count * sizeof(int), MALLOC_CAP_INTERNAL);

    if(arr == nullptr) {
        // Gestion propre si OOM
        Serial.println("OOM — skip");
        return;
    }

    for(size_t i = 0; i < count; i++) arr[i] = i;
    processArray(arr, count);

    free(arr); // Libération indispensable
}

📊 Impact mesuré :

  • Heap non optimisé : ~120 mA → 2,22 kWh/an
  • Heap optimisé : ~80 mA → 1,48 kWh/an
  • Économie : 0,74 kWh/an → 0,42 kg CO₂/an
  • 🚗 Équivalent voiture : ~7 km non parcourus
  • 🔧 Bonus : élimination des reboots intempestifs
6

Désactiver les modules et périphériques non utilisés

#wifi #bluetooth #périphériques Intermédiaire
−72%
Consommation
~22 km
voiture/an
⭐⭐⭐⭐⭐

Pourquoi ? Par défaut sur l'ESP32, le Wi-Fi, le Bluetooth et le convertisseur analogique-numérique (ADC) sont actifs même si on ne les utilise pas. Le Wi-Fi seul consomme ~80 mA en écoute passive. Sur un capteur IoT qui envoie une mesure toutes les 10 minutes, garder le Wi-Fi constamment actif gaspille plus de 99% de son énergie réseau.

❌ Anti-pattern : modules actifs par défaut

// Mauvais : Wi-Fi et BT actifs même sans utilisation
void setup() {
    M5.begin(); // Wi-Fi et BT initialisés et actifs → ~180 mA
}

void loop() {
    readSensor(); // Pas besoin de Wi-Fi ici !
    delay(1000);
}

✅ Solution : désactivation explicite + activation à la demande

// Optimisé : tout éteint par défaut, allumé à la demande
void setup() {
    M5.begin();

    WiFi.mode(WIFI_OFF);      // Désactiver Wi-Fi : −80 mA
    btStop();                  // Désactiver Bluetooth : −30 mA
    adc_power_release();       // Désactiver ADC : −5 mA
    esp_wifi_stop();           // Arrêt complet driver Wi-Fi
}

void sendData(const char* payload) {
    WiFi.mode(WIFI_STA);      // Activer uniquement pour l'envoi
    WiFi.begin(SSID, PASS);
    while(WiFi.status() != WL_CONNECTED) delay(100);
    httpPost(payload);
    WiFi.disconnect(true);
    WiFi.mode(WIFI_OFF);       // Rééteindre immédiatement
}

📊 Impact mesuré :

  • Wi-Fi/BT actifs : ~180 mA → 4,21 kWh/an
  • Modules désactivés : ~50 mA → 1,62 kWh/an
  • Économie : 2,59 kWh/an → ~1,49 kg CO₂/an
  • 🚗 Équivalent voiture : ~22 km non parcourus
  • 🔋 Autonomie batterie (700 mAh) : 3,9h → 14h
7

Privilégier constexpr et const pour les valeurs fixes

#rom #compilateur #optimisation Intermédiaire
−2 Ko
RAM économisée
+10%
Performance
⭐⭐⭐

Pourquoi ? Une variable globale non-const est stockée en RAM (DRAM, ~150 mA d'accès). Avec constexpr, le compilateur résout la valeur à la compilation et la stocke en Flash (ROM, lecture moins énergivore). De plus, les constantes permettent au compilateur d'appliquer des optimisations comme le constant folding ou la suppression de branches mortes.

❌ Anti-pattern : variables en RAM

// Mauvais : stocké en RAM, pas optimisable
float PI = 3.14159f;
int   MAX_BUFFER = 512;
char  DEVICE_NAME[] = "Sensor-01"; // Copié en RAM au démarrage

✅ Solution : constantes en ROM

// Optimisé : résolu à la compilation, stocké en Flash
constexpr float PI         = 3.14159f;
constexpr int   MAX_BUFFER  = 512;
constexpr int   BAUD_RATE   = 115200;

// Chaînes en Flash (PROGMEM sur AVR, RODATA sur ESP32)
const char DEVICE_NAME[] PROGMEM = "Sensor-01";

// Macro pour les littéraux souvent réutilisés
#define SCREEN_W 320
#define SCREEN_H 240

📊 Impact mesuré :

  • RAM économisée : ~2 Ko (pour un firmware typique)
  • Gain performance CPU : +10% (constant folding)
  • Économie courant : ~10 mA → 0,32 kWh/an → 0,18 kg CO₂/an
  • 🚗 Équivalent voiture : ~1,6 km non parcourus
8

Adapter dynamiquement la fréquence CPU aux besoins

#dvfs #cpu-freq #énergie Avancé
−44%
Consommation CPU
~12 km
voiture/an
⭐⭐⭐⭐

Pourquoi ? L'ESP32 peut fonctionner à 80, 160 ou 240 MHz. La consommation du CPU est approximativement proportionnelle à f × V² (fréquence × carré de la tension). Passer de 240 à 80 MHz réduit la consommation CPU d'environ 44% pour les tâches légères (I/O, attentes, traitement de capteurs simples), sans impact sur l'expérience utilisateur.

❌ Anti-pattern : fréquence maximale permanente

// Mauvais : 240 MHz tout le temps, même pour lire un capteur
void setup() {
    // Fréquence par défaut : 240 MHz → ~160 mA
}

void loop() {
    readTemperature(); // Tâche légère, 240 MHz inutile
    delay(5000);
}

✅ Solution : DVFS — ajustement dynamique

// Optimisé : fréquence adaptée à la charge
void setup() {
    setCpuFrequencyMhz(80); // Base : 80 MHz → ~90 mA
}

void runHeavyComputation() {
    setCpuFrequencyMhz(240); // Monter si besoin
    performFFT();
    setCpuFrequencyMhz(80);  // Redescendre immédiatement
}

void loop() {
    readTemperature();          // 80 MHz suffit
    if(heavyTaskNeeded) runHeavyComputation();
    delay(5000);
}

📊 Impact mesuré :

  • 240 MHz permanent : ~160 mA → 2,97 kWh/an
  • 80 MHz adaptatif : ~90 mA → 1,67 kWh/an
  • Économie : 1,30 kWh/an → 0,74 kg CO₂/an
  • 🚗 Équivalent voiture : ~12 km non parcourus
9

Minimiser les accès à la mémoire Flash — cache en RAM

#flash #cache #spiffs Avancé
−65%
Consommation Flash
~22 km
voiture/an
⭐⭐⭐⭐⭐

Pourquoi ? Un accès à la mémoire Flash SPIFFS ou une carte SD consomme ~200 mA pendant la lecture (activation du driver SPI + Flash). Lire un fichier de configuration à chaque itération de la boucle peut représenter 90% de la consommation totale, alors qu'une lecture unique au démarrage avec mise en cache en RAM ramène cette opération à ~70 mA.

❌ Anti-pattern : lecture Flash dans la boucle principale

// Mauvais : accès Flash à chaque itération → ~200 mA par lecture
void loop() {
    File f = SPIFFS.open("/config.json", "r");
    String config = f.readString(); // Réveille le driver Flash
    f.close();
    applyConfig(config);
    delay(1000);
}

✅ Solution : lecture unique + cache en RAM

// Optimisé : lecture au démarrage, accès RAM ensuite
String configCache = "";

void setup() {
    SPIFFS.begin(true);
    File f = SPIFFS.open("/config.json", "r");
    configCache = f.readString(); // Une seule lecture Flash
    f.close();
    SPIFFS.end(); // Éteindre le driver Flash
}

void loop() {
    applyConfig(configCache); // Accès RAM : ~70 mA, instantané
    delay(1000);
}

📊 Impact mesuré :

  • Lectures Flash répétées : ~200 mA → 3,71 kWh/an
  • Cache RAM : ~70 mA → 1,30 kWh/an
  • Économie : 2,41 kWh/an → 1,38 kg CO₂/an
  • 🚗 Équivalent voiture : ~22 km non parcourus
  • ⚡ Bonus : durée de vie Flash prolongée (cycles d'écriture limités)
10

Choisir les types de données les plus petits adaptés au cas d'usage

#types #mémoire #cache Débutant
−60%
Mémoire
~2,5 km
voiture/an
⭐⭐⭐

Pourquoi ? Sur un ESP32 avec 520 Ko de RAM, utiliser long (8 octets) pour un compteur qui n'atteindra jamais 255 gaspille 7 octets. Multiplié par des milliers de variables dans un firmware complexe, c'est plusieurs Ko de RAM perdus, ce qui augmente les accès Flash, réduit l'efficacité du cache CPU et consomme davantage d'énergie.

❌ Anti-pattern : types surdimensionnés

// Mauvais : tous les types sont "larges"
long   counter     = 0;        // 8 octets pour 0–100
double temperature = 25.5;     // 8 octets, précision inutile
int    flags[8];               // 32 octets pour 8 booléens

✅ Solution : types adaptés + bitfields

// Optimisé : types minimaux
uint8_t  counter     = 0;       // 1 octet suffit (0–255)
float    temperature = 25.5f;   // 4 octets, précision suffisante

// Bitfield pour drapeaux : 8 booléens en 1 octet
struct DeviceFlags {
    uint8_t wifiEnabled  : 1;
    uint8_t displayOn    : 1;
    uint8_t sensorReady  : 1;
    uint8_t dataUploaded : 1;
    uint8_t reserved     : 4;
} flags; // 1 seul octet !

📊 Impact mesuré :

  • Mémoire économisée : ~60% sur les structures de données
  • Performance cache CPU : +15% (meilleure localité des données)
  • Économie courant : ~15 mA → 0,49 kWh/an → 0,28 kg CO₂/an
  • 🚗 Équivalent voiture : ~2,5 km non parcourus
