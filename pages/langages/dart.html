Dart - Éco-conception & Green Coding

PHP Dart Python (bientôt) C/C++ (bientôt) Java (bientôt) JavaScript (bientôt)

Dart, le langage derrière Flutter, offre un positionnement unique en termes d'efficacité énergétique. Bien que classé parmi les langages interprétés gourmands en mémoire (570 Mo en moyenne), sa compilation AOT (Ahead-of-Time) en production permet d'atteindre des performances proches du natif tout en maintenant un développement agile avec le mode JIT (Just-In-Time).

Flutter permet de créer des applications cross-platform performantes (Web, Android, iOS) avec une base de widgets réutilisables (liste à puce, input de texte, boutons, etc.). Ce framework est connu pour avoir un toolkit pour le développement rapide de jeux et d'applications interactives (allant des jeux de carte jusqu'aux platformers et jeux 2D complexes). Cependant, sans bonnes pratiques d'éco-conception, les applications Dart/Flutter peuvent rapidement devenir gourmandes en ressources, impactant l'environnement.


Cette page présente 10 règles d'éco-conception spécifiques à Dart et Flutter pour réduire l'empreinte environnementale de vos applications cross-platform, avec des exemples concrets et des mesures de gains réels en performance et consommation énergétique.

Les 10 Éco-Rules Dart/Flutter

1 const_widgets - Utilisation systématique de const

Impact TRÈS ÉLEVÉ

Code inefficace :

// Crée un nouvel objet à chaque reconstruction d'interface
return Padding(
  padding: EdgeInsets.all(8.0), 
  child: Text("Bonjour"),
);

Pourquoi c'est inefficace ?

  • Flutter recrée les widgets à chaque rebuild sans const
  • Augmente le travail du Garbage Collector
  • Consomme plus de CPU et mémoire inutilement

Code optimisé :

// L'instance est unique et réutilisée en mémoire
return const Padding(
  padding: EdgeInsets.all(8.0),
  child: Text("Bonjour"),
);

Gains mesurés :

  • Réduction drastique des cycles CPU lors des rebuilds
  • Diminution de 40-60% de l'usage RAM [4]
  • Instance canonique créée une seule fois en mémoire

2 listview_builder - Lazy Loading avec ListView.builder

Impact CRITIQUE

Code inefficace :

// Charge 10 000 éléments en mémoire immédiatement
ListView(
  children: items.map((i) => MyWidget(i)).toList(),
)

Pourquoi c'est inefficace ?

  • Crée tous les widgets d'un coup, même non visibles
  • Consommation mémoire exponentielle avec le nombre d'items
  • Temps de chargement initial très long

Code optimisé :

// Ne crée que les widgets visibles à l'écran
ListView.builder(
  itemCount: items.length,
  itemBuilder: (context, index) => MyWidget(items[index]),
)

Gains mesurés :

  • ×10 plus rapide pour le chargement initial
  • Économie de 80-90% de mémoire vive [5]
  • Création à la demande uniquement des widgets visibles

3 stringbuffer - StringBuffer pour concaténations

Impact ÉLEVÉ

Code inefficace :

String s = "";
for (var i = 0; i < 100; i++) {
  s += "donnée $i"; // Crée un nouvel objet String à chaque fois
}

Pourquoi c'est inefficace ?

  • Chaque += crée une nouvelle chaîne en mémoire
  • Pour 100 itérations : 100 allocations distinctes
  • Surcharge importante du Garbage Collector

Code optimisé :

var buffer = StringBuffer();
for (var i = 0; i < 100; i++) {
  buffer.write("donnée $i");
}
String s = buffer.toString(); // Une seule allocation finale

Gains mesurés :

  • ×3-4 plus rapide selon la taille [6]
  • Réduction de 50-70% de l'usage mémoire
  • Une seule allocation au lieu de N

4 aot_compilation - Activer la compilation AOT

Impact CRITIQUE

Configuration inefficace :

// Mode Debug avec JIT - bon pour le développement
flutter run

// Mais trop lent et gourmand pour la production

Pourquoi c'est inefficace ?

  • Le mode JIT (Just-In-Time) compile à la volée
  • Overhead de la machine virtuelle Dart
  • Temps de démarrage plus long et consommation CPU élevée

Configuration optimisée :

// Build Release avec compilation AOT (Ahead-of-Time)
flutter build apk --release
flutter build ios --release

// Avec optimisations supplémentaires
flutter build apk --release --split-per-abi
flutter build apk --release --obfuscate --split-debug-info=build/symbols

Gains mesurés :

  • +30% de vitesse d'exécution vs JIT [3]
  • Temps de démarrage réduit de 40-50%
  • Code compilé en machine natif = moins de CPU

5 async_programming - Programmation asynchrone efficace

Impact ÉLEVÉ

Code inefficace :

// Bloque le thread UI principal
void loadData() {
  var data = fetchFromApi(); // Bloquant!
  setState(() {
    _data = data;
  });
}

Pourquoi c'est inefficace ?

  • Le thread UI est bloqué pendant le chargement
  • Interface gelée = mauvaise expérience utilisateur
  • CPU occupé en attente au lieu d'être idle

Code optimisé :

// Utilise async/await pour libérer le thread
Future<void> loadData() async {
  var data = await fetchFromApi(); // Non-bloquant!
  setState(() {
    _data = data;
  });
}

// Ou avec FutureBuilder
FutureBuilder(
  future: fetchFromApi(),
  builder: (context, snapshot) => ...
)

Gains mesurés :

  • UI fluide même pendant les opérations lourdes
  • Réduction de la consommation CPU idle [7]
  • Meilleure expérience utilisateur et réactivité

6 build_method_pure - Méthode build() pure et légère

Impact ÉLEVÉ

Code inefficace :

@override
Widget build(BuildContext context) {
  // Calculs lourds dans build() - MAUVAIS!
  var processedData = expensiveComputation(data);
  var filteredList = items.where((i) => i > 10).toList();
  
  return Text(processedData);
}

Pourquoi c'est inefficace ?

  • build() peut être appelé jusqu'à 120 fois/seconde
  • Calculs répétés inutilement à chaque frame
  • Surcharge CPU et risque de janks (frames perdus)

Code optimisé :

// Calculs dans initState() ou didUpdateWidget()
@override
void initState() {
  super.initState();
  _processedData = expensiveComputation(widget.data);
  _filteredList = widget.items.where((i) => i > 10).toList();
}

@override
Widget build(BuildContext context) {
  // build() reste pure et rapide
  return Text(_processedData);
}

Gains mesurés :

  • Framerate stable à 60 FPS
  • Réduction de 50% du temps build() [8]
  • Moins de travail pour le GC

7 image_optimization - Optimisation des images

Impact ÉLEVÉ

Code inefficace :

// Image haute résolution non optimisée
Image.network(
  'https://example.com/huge-image-4K.jpg', // 8 MB!
)

Pourquoi c'est inefficace ?

  • Téléchargement de plusieurs MB de données
  • Décodage image = forte consommation CPU
  • Mémoire vive saturée par les images en cache

Code optimisé :

// Utiliser CachedNetworkImage avec dimensions
CachedNetworkImage(
  imageUrl: 'https://example.com/optimized.webp',
  width: 200,
  height: 200,
  memCacheWidth: 200,
  memCacheHeight: 200,
  placeholder: (context, url) => CircularProgressIndicator(),
  errorWidget: (context, url, error) => Icon(Icons.error),
)

Gains mesurés :

  • Réduction de 60-80% du poids des images (WebP)
  • Cache automatique = moins de requêtes réseau [9]
  • Décodage optimisé avec dimensions cibles

8 tree_shaking - Tree-shaking et code splitting

Impact MOYEN-ÉLEVÉ

Configuration inefficace :

// Import de bibliothèques entières
import 'package:my_app/all_features.dart';

// Tout le code est inclus dans le bundle final

Pourquoi c'est inefficace ?

  • Bundle final gonflé avec du code inutilisé
  • Temps de téléchargement et startup plus longs
  • Mémoire occupée par du code mort

Configuration optimisée :

// Import spécifique uniquement
import 'package:my_app/features/login.dart';

// Avec deferred loading pour code splitting
import 'package:my_app/features/admin.dart' deferred as admin;

// Charger à la demande
await admin.loadLibrary();
admin.showAdminPanel();

Gains mesurés :

  • Réduction de 40-50% de la taille du bundle [10]
  • Startup plus rapide de 30-40%
  • Tree-shaking automatique élimine le code mort

9 stateless_first - Préférer StatelessWidget

Impact MOYEN

Code inefficace :

// StatefulWidget pour un contenu statique
class MyLabel extends StatefulWidget {
  @override
  _MyLabelState createState() => _MyLabelState();
}

class _MyLabelState extends State<MyLabel> {
  @override
  Widget build(BuildContext context) {
    return Text('Label fixe');
  }
}

Pourquoi c'est inefficace ?

  • StatefulWidget crée un objet State supplémentaire
  • Overhead de gestion du cycle de vie
  • Consommation mémoire et CPU inutile

Code optimisé :

// StatelessWidget pour contenu statique
class MyLabel extends StatelessWidget {
  const MyLabel();

  @override
  Widget build(BuildContext context) {
    return const Text('Label fixe');
  }
}

Gains mesurés :

  • Moins d'objets en mémoire
  • Rebuild plus rapide [11]
  • Code plus simple et maintenable

10 dispose_resources - Libérer les ressources proprement

Impact ÉLEVÉ

Code dangereux :

class _MyWidgetState extends State<MyWidget> {
  StreamSubscription? _subscription;
  AnimationController? _controller;

  @override
  void initState() {
    super.initState();
    _subscription = stream.listen((data) => ...);
    _controller = AnimationController(...);
  }
  
  // ⚠️ Pas de dispose() = fuite mémoire!
}

Pourquoi c'est dangereux ?

  • Les ressources ne sont jamais libérées
  • Fuites mémoire progressives
  • Listeners actifs continuent de consommer CPU

Code optimisé :

class _MyWidgetState extends State<MyWidget> {
  StreamSubscription? _subscription;
  AnimationController? _controller;

  @override
  void initState() {
    super.initState();
    _subscription = stream.listen((data) => ...);
    _controller = AnimationController(...);
  }

  @override
  void dispose() {
    // ✅ Libération propre des ressources
    _subscription?.cancel();
    _controller?.dispose();
    super.dispose();
  }
}

Gains mesurés :

  • Évite les fuites mémoire
  • Libère CPU des listeners inactifs [12]
  • Application plus stable dans la durée
