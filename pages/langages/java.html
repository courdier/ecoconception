Les 10 Éco-Rules PHP

1 StringBuilder - Concaténation de chaînes en boucle

Impact ÉLEVÉ

Code inefficace :

String result = "";
for (String item : items) {
    result += item + ", ";
}
result = result.substring(0, result.length() - 2);

Pourquoi c'est inefficace ?

  • En Java, String est immuable : chaque += crée une nouvelle chaîne en mémoire
  • Pour 1000 éléments : 1000 allocations mémoire et copies (O(n²))
  • Surconsommation mémoire CPU + temps d'exécution exponentiel

Code optimisé :

StringBuilder result = new StringBuilder();
for (int i = 0; i < items.size(); i++) {
    result.append(items.get(i));
    if (i < items.size() - 1) result.append(", ");
}
String finalResult = result.toString();

Gains mesurés :

  • ×10 plus rapide pour 10 000 éléments [3]
  • Réduit la consommation mémoire de 70-90%
  • Complexité O(n) au lieu de O(n²)

2 Stream Filtering - Filtrage efficace de collections

Impact ÉLEVÉ

Code inefficace :

List<User> activeUsers = new ArrayList<>();
for (User user : users) {
    if (user.isActive()) {
        activeUsers.add(user);
    }
}

Pourquoi c'est inefficace ?

  • Code impératif avec boucle explicite et mutations
  • Crée une liste intermédiaire même si non nécessaire
  • Moins optimisable par le JIT compiler

Code optimisé :

List<User> activeUsers = users.stream()
    .filter(User::isActive)
    .collect(Collectors.toList());

Gains mesurés :

  • ×2-3 plus rapide avec parallel streams [4]
  • Réduit la consommation mémoire de 40-60% (lazy evaluation)
  • Code plus lisible et facilement parallélisable

3 HashMap vs List - Recherche optimale

Impact TRÈS ÉLEVÉ

Code inefficace :

List<Permission> permissions = new ArrayList<>();
permissions.add(new Permission("READ"));
permissions.add(new Permission("WRITE"));

if (permissions.stream().anyMatch(p -> p.getName().equals("READ"))) {
    // Accès autorisé
}

Pourquoi c'est inefficace ?

  • List.contains() parcourt séquentiellement (O(n))
  • Pour 10 000 permissions : ×100 plus lent qu'un HashMap [5]
  • Temps CPU croît linéairement avec la taille de la collection

Code optimisé :

Set<String> permissions = Set.of("READ", "WRITE");

if (permissions.contains("READ")) {
    // Accès autorisé
}

Gains mesurés :

  • Performance : O(1) vs O(n), temps constant [5]
  • Pour 100k éléments : ×1000+ plus rapide
  • Complexité constante peu importe la taille

4 Lazy Initialization - Initialisation tardive

Impact TRÈS ÉLEVÉ

Code inefficace :

public class DatabaseCache {
    private List<User> users;
    
    public DatabaseCache() {
        // Charge 1M de users au démarrage !
        this.users = loadAllUsersFromDB();
    }
}

Pourquoi c'est inefficace ?

  • Charge tous les données même si non utilisées
  • Temps de démarrage long + consommation mémoire massive
  • Gaspille de l'énergie pour les données jamais accédées

Code optimisé :

public class DatabaseCache {
    private List<User> users = null;
    
    public synchronized List<User> getUsers() {
        if (users == null) {
            users = loadAllUsersFromDB();
        }
        return users;
    }
}

Gains mesurés :

  • Temps de démarrage réduit de 50-80%
  • Mémoire initiale : quasi zéro (chargement on-demand)
  • Ne charge que ce qui est réellement utilisé

5 Object Pooling - Réutilisation d'objets

Impact ÉLEVÉ

Code inefficace :

public void processRequests(List<Request> requests) {
    for (Request req : requests) {
        // Créé 10k objets Buffer !
        Buffer buffer = new Buffer(1024);
        process(req, buffer);
    }
}

Pourquoi c'est inefficace ?

  • Crée 10 000 objets Buffer lors d'une boucle
  • Chaque création = allocation mémoire + GC overhead
  • Garbage Collector doit nettoyer 10k allocations inutiles = consommation CPU/énergie

Code optimisé :

public void processRequests(List<Request> requests) {
    Buffer buffer = new Buffer(1024);
    for (Request req : requests) {
        buffer.clear();  // Réutilise l'objet
        process(req, buffer);
    }
}

Gains mesurés :

  • ×3-5 plus rapide pour les boucles intensives [6]
  • Réduit mémoire de 60-80% (1 objet au lieu de 10k)
  • GC pause time diminue drastiquement

6 Parallel Streams - Parallélisation des traitements

Impact TRÈS ÉLEVÉ

Code inefficace :

List<Integer> results = numbers.stream()
    .map(n -> complexCalculation(n))
    .collect(Collectors.toList());
// Traitement séquentiel = 100% d'un seul core

Pourquoi c'est inefficace ?

  • Traitement séquentiel : n'utilise qu'un seul core CPU
  • Autres cores restent inactifs = gaspillage de ressources
  • Sur une machine 8-core : 87.5% des ressources perdues

Code optimisé :

List<Integer> results = numbers.parallelStream()
    .map(n -> complexCalculation(n))
    .collect(Collectors.toList());
// Traitement parallèle = utilise tous les cores !

Gains mesurés :

  • ×4 plus rapide sur 8 cores [7]
  • Temps total réduit = moins d'énergie consommée
  • Meilleure utilisation des ressources matérielles

7 GC Tuning - Optimiser le Garbage Collector

Impact CRITIQUE

Configuration inefficace :

// Lancer Java avec GC par défaut
java -jar app.jar

// GC standard = stop-the-world pauses longues !

Pourquoi c'est inefficace ?

  • GC par défaut provoque des pauses stop-the-world longues
  • L'application s'arrête pendant que GC nettoie la mémoire
  • Temps gaspillé = CPU idle = énergie consommée inutilement

Configuration optimisée :

// Utiliser G1GC (meilleur pour applications modernes)
java -XX:+UseG1GC \
     -XX:MaxGCPauseMillis=200 \
     -XX:InitiatingHeapOccupancyPercent=35 \
     -jar app.jar

Gains mesurés :

  • ×2-3 plus rapide avec G1GC optimisé [8]
  • GC pauses réduites de 50-70%
  • Consommation énergie directement proportionnelle au temps d'exécution

8 Try-with-Resources - Gestion propre des ressources

Impact MOYEN

Code dangereux :

FileInputStream fis = new FileInputStream("file.txt");
try {
    byte[] data = fis.readAllBytes();
    // ⚠️ Si exception avant finally : ressource NON libérée
} finally {
    fis.close();
}

Pourquoi c'est dangereux ?

  • Possibilité de fuite de ressources (fichiers ouverts, connexions DB...)
  • Code boilerplate inefficace et sujet aux bugs
  • Ressources orphelines consomment de la mémoire

Code optimisé :

try (FileInputStream fis = new FileInputStream("file.txt")) {
    byte[] data = fis.readAllBytes();
    // ✅ Ressource automatiquement fermée
} catch (IOException e) {
    e.printStackTrace();
}

Gains mesurés :

  • Évite les fuites de ressources (fichiers, sockets...)
  • Code plus lisible et moins sujet aux erreurs [9]
  • Libération garantie même en cas d'exception

9 Null Checks - Vérifications ciblées

Impact FAIBLE

Code inefficace :

if (config != null && config.getDatabase() != null 
    && config.getDatabase().getHost() != null) {
    connect(config.getDatabase().getHost());
}

Pourquoi c'est inefficace ?

  • Vérifications répétées de null = overhead CPU
  • Code difficile à lire et maintenir
  • Appels de méthode répétés pour chaque vérification

Code optimisé :

String host = Optional.ofNullable(config)
    .map(Config::getDatabase)
    .map(Database::getHost)
    .orElse(null);

if (host != null) {
    connect(host);
}

Gains mesurés :

  • ×1.05-1.08 plus rapide avec Optional (lazy evaluation) [10]
  • Moins d'appels de méthode
  • Code plus lisible et idiomatique

10 Dependency Injection - Injection de dépendances

Impact MOYEN

Code inefficace :

public class UserService {
    private DatabaseConnection db;
    
    public UserService() {
        // Création directe = couplage fort
        this.db = new DatabaseConnection("prod");
    }
}

Pourquoi c'est inefficace ?

  • Couplage fort : crée toujours une instance réelle
  • Impossible de tester ou d'utiliser des mocks/stubs légers
  • Charge de ressources inutiles au démarrage

Code optimisé :

public class UserService {
    private final DatabaseConnection db;
    
    public UserService(DatabaseConnection db) {
        // ✅ Injection via constructeur = couplage faible
        this.db = db;
    }
}

// Test : injection facile d'un mock
UserService service = new UserService(new MockDatabaseConnection());

Gains mesurés :

  • Meilleure testabilité = moins de code à refactoriser
  • Flexibilité : utiliser des stubs/mocks pour tests (très légers)
  • Réduction de la charge au démarrage avec lazy initialization [11]

Bonnes pratiques complémentaires

Type Safety

public long calculateTotal(double price, int qty) {
    return (long) (price * qty);
}

Types explicites = moins de conversions runtime → économie CPU

Requêtes SQL

// ✅ Une seule requête avec IN
List<User> users = db.query(
    "SELECT * FROM users WHERE id IN ?", userIds
);

Évite le problème N+1 → moins de requêtes réseau

Iterator Pattern

try (ResultSet rs = stmt.executeQuery()) {
    while (rs.next()) {
        process(rs);  // Itération sans charger tout en mémoire
    }
}

Mémoire constante au lieu de charger 100k lignes
