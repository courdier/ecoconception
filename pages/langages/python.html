Python - Éco-conception & Green Coding

PHP Rust Python C/C++ Java Dart JavaScript (bientôt)

Python est un langage de programmation interprété, polyvalent et largement utilisé dans de nombreux domaines. Sa simplicité de syntaxe permet de développer rapidement, mais peut conduire à des mauvaises pratiques ayant un impact environnemental significatif.


Cette page présente 10 règles d'éco-conception concrètes pour réduire l'empreinte carbone de vos applications Python, avec des mesures réelles via CodeCarbon et des équivalences écologiques parlantes basées sur les métriques Green IT.

Set vs List pour la recherche

#structures-de-donnees #algorithmique #cpu
97.0%
Gain CO₂
2,207×
Plus rapide
⭐⭐⭐⭐⭐

Cas d'usage : Vérification d'existence dans une collection (authentification, permissions), filtrage et dédoublonnage de données, recherche fréquente dans des référentiels (codes postaux, IDs produits).

Principe

Utiliser des structures à accès constant O(1) (set, dict) plutôt que linéaire O(n) (list) pour les opérations de recherche fréquentes.

Anti-pattern (O(n) - linéaire) : 

# Recherche séquentielle dans une liste
items_list = list(range(1_000_000))
target = 999_999  # Pire cas : dernier élément
found = target in items_list  # O(n) - parcourt toute la liste

Sol ution (O(1) - constant) : 

# Recherche par hash dans un set
items_set = set(range(1_000_000))
found = target in items_set  # O(1) - accès direct par table de hachage

Explication

  • Liste : Stockage séquentiel, recherche linéaire O(n) — parcourt potentiellement tout le tableau
  • Set : Table de hachage, calcul d'index direct O(1) — accès direct via hash
  • Impact : Pour 1M éléments avec 4,000 recherches, passage de ~39 secondes à ~0.018 secondes

Résultats mesurés :

Métrique Liste Set Gain
Temps 38.967 s 0.018 s 2,207×
Mémoire 0.3 Mo 0.3 Mo Similaire
Énergie 0.0000563 kWh 0.0000017 kWh 97.0%
CO₂ 0.0323 g 0.0010 g 97.0%

Équivalence concrète (Green IT) :

Pour un système traitant 1 milliard de requêtes de recherche par jour :

  • Économie CO₂ : 11,372 kg CO₂/an (équivalent à 101,536 km en voiture selon ADEME)
  • Équivalence française : 27 fois l'empreinte numérique annuelle d'un Français (420 kg CO₂)
  • Énergie : 19,880 kWh/an (équivalent à la fabrication de 353 téléphones)
  • Infrastructure : Réduction significative des serveurs nécessaires

Générateurs pour flux massifs

#memoire #streaming #scalabilite
100.0%
Gain Mémoire
41.1%
Gain CO₂
⭐⭐⭐⭐

Cas d'usage : Traitement de fichiers logs volumineux (GB/TB), streaming de données depuis des bases de données, pipeline ETL (Extract, Transform, Load) sur big data, lecture de fichiers CSV/JSON trop grands pour la RAM.

Principe

Utiliser des générateurs (yield, expressions génératrices) pour traiter les données en streaming, évitant le chargement complet en mémoire. Le gain principal est la stabilité mémoire (pas de risque OOM).

Anti-pattern : 

# Chargement complet en mémoire - Risque OOM
def lire_flux_donnees():
    for i in range(20_000_000):
        yield str(i)

def transformer(ligne):
    return ligne + "_traitee"

# Étape 1 : On stocke tout dans une liste (consomme énormément de RAM)
lignes = list(lire_flux_donnees())
# Étape 2 : On traite la liste
resultats = [transformer(l) for l in lignes]
total = sum(len(r) for r in resultats)

Solution : 

# Traitement streaming - O(1) mémoire
# On traite un élément à la fois : un seul objet est en RAM à l'instant T
total_gen = sum(len(transformer(ligne)) for ligne in lire_flux_donnees())

Explication

  • Liste : Allocation O(n), pic mémoire proportionnel à la taille des données (2.3 Go pour 20M éléments)
  • Générateur : Allocation O(1), traitement à la volée (0.3 Mo constant)
  • Bénéfice écologique indirect : Évite le swapping disque (très énergivore) et les crashs OOM

Résultats mesurés :

Métrique Liste Générateur Gain
Temps 46.905 s 31.174 s 1.5×
Mémoire 2,307.1 Mo 0.3 Mo 100.0%
Énergie 0.0000835 kWh 0.0000492 kWh 41.1%
CO₂ 0.0478 g 0.0282 g 41.1%

Pourquoi c'est efficace ?

Le gain énergétique direct est modeste (41.1%), mais la réduction mémoire de 100.0% évite :

  • Le swapping disque (très énergivore)
  • Les crashs OOM et redémarrages serveur
  • Le surdimensionnement des infrastructures

Équivalence concrète (Green IT) :

Pour un fichier journal de 100 Go traité quotidiennement :

  • Économie mémoire : 99.9 Go de RAM non alloués
  • Évite le swapping : ~500 kg CO₂/an économisés
  • Équivalence : 2 serveurs virtuels en moins

NumPy pour calcul numérique

#calcul-numerique #vectorisation #ml
97.0%
Gain CO₂
594×
Plus rapide
⭐⭐⭐⭐⭐

Cas d'usage : Calcul scientifique, machine learning, data science, traitement d'images et signaux, simulations numériques (Monte Carlo, modèles physiques), analyse financière sur séries temporelles.

Principe

Préférer NumPy aux listes Python pour stocker des collections homogènes. NumPy utilise de la mémoire contiguë et des opérations vectorisées en C, évitant l'overhead des objets Python.

Anti-pattern (Python pur) : 

# Liste de flottants - stockage dispersé, opération lente
numbers_list = [float(i) for i in range(10_000_000)]
result = sum([x**2 for x in numbers_list])

Solution (NumPy vectorisé) : 

import numpy as np

# Tableau contigu en mémoire, vectorisé (C-style)
numbers_numpy = np.arange(10_000_000, dtype=np.float64)
# Opération vectorisée exécutée en C (très basse consommation)
result = np.sum(numbers_numpy**2)  # BLAS/LAPACK optimisé

Explication

  • Liste Python : Tableau de pointeurs vers objets Float, overhead ~28 octets par nombre, stockage dispersé
  • NumPy : Buffer contigu de valeurs brutes (8 octets par double), SIMD possible, meilleure localité du cache
  • Vectorisation : Délégation aux bibliothèques C optimisées (BLAS/LAPACK)

Résultats mesurés :

Mémoire 627.8 Mo 152.6 Mo 75.7%
Énergie 0.0000600 kWh 0.0000018 kWh 97.0%
CO₂ 0.0344 g 0.0010 g 97.0%

Équivalence concrète (Green IT) :

Pour un modèle de machine learning traitant 1 million d'échantillons × 1,000 caractéristiques :

  • Économie CO₂ : 30 kg CO₂/entraînement (équivalent à 268 km en voiture selon ADEME)
  • Équivalence française : ~7% de l'empreinte numérique annuelle d'un Français
  • Temps de calcul : Réduction de 6 heures à 36 secondes
  • Infrastructure : 1 serveur au lieu de 10 pour le même débit

Concaténation de chaînes optimisée

#strings #web #json #memoire
93.9%
Gain CO₂
58.3×
Plus rapide
⭐⭐⭐⭐⭐

Cas d'usage : Génération de pages HTML, XML, JSON dynamiques, construction de requêtes SQL ou de logs, sérialisation de données en masse, templates et génération de rapports.

Principe

Utiliser str.join() ou io.StringIO plutôt que l'opérateur += pour construire des chaînes dans des boucles.

Anti-pattern (O(n²)) : 

# Concaténation quadratique O(n²) - Nouvelle allocation + copie!
elements = [f"item_{i}" for i in range(50_000)]
html = ""
for item in elements:
    html += f"<div>{item}</div>"  # Inefficace : crée une nouvelle chaîne à chaque fois

Solution (O(n)) : 

# Solution pythonique avec join - une seule allocation
# Efficace : calcule la taille totale et alloue la mémoire une seule fois
html = "".join(f"<div>{item}</div>" for item in elements)

Explication

  • += sur str : Immuable → nouvelle allocation + copie complète → O(n²)
  • join : Calcule taille totale upfront → une seule allocation → O(n)

Résultats mesurés :

Métrique += join() Gain
Temps 37.770 s 0.648 s 58.3×
Mémoire 2.1 Mo 5.4 Mo -155.3%
Énergie 0.0000607 kWh 0.0000037 kWh 93.9%
CO₂ 0.0348 g 0.0021 g 93.9%

Équivalence concrète (Green IT) :

Pour un site web générant 1 million de pages HTML par jour :

  • Économie CO₂ : 119 kg CO₂/an (équivalent à 1,062 km en voiture selon ADEME)
  • Équivalence française : ~28% de l'empreinte numérique annuelle d'un Français
  • Temps de génération : Réduction de 11 heures à 11 minutes quotidiennes
  • Ressources : 94% moins de CPU utilisé pour le même service

Bufferisation I/O

#io #disque #fichiers #systeme
98.4%
Gain CO₂
235.3×
Plus rapide
⭐⭐⭐⭐⭐

Cas d'usage : Copie de fichiers volumineux, traitement de logs en temps réel, sauvegarde et restauration de bases de données, streaming vidéo/audio.

Principe

Lire et écrire par blocs de taille optimale (4-64 KB) plutôt qu'octet par octet.

Anti-pattern : 

# 1 octet = 1 appel système - catastrophe!
with open(TEST_FILE, "rb") as f_in, open(DEST_SLOW, "wb") as f_out:
    while True:
        octet = f_in.read(1)  # Un appel système par octet : désastreux
        if not octet:
            break
        f_out.write(octet)

Solution : 

# 64KB = 1 appel système - optimal!
BUFFER_SIZE = 64 * 1024  # 64 Ko : taille optimale pour de nombreux systèmes
with open(TEST_FILE, "rb") as f_in, open(DEST_FAST, "wb") as f_out:
    while True:
        bloc = f_in.read(BUFFER_SIZE)
        if not bloc:
            break
        f_out.write(bloc)

Explication

  • Appel système : Transition user/kernel coûteuse (~1000 cycles CPU)
  • Buffer optimal : 64KB correspond à la taille de page mémoire et secteur disque

Résultats mesurés :

Métrique 1 octet 64KB Gain
Temps 150.701 s 0.640 s 235.3×
Mémoire 0.29 Mo 0.33 Mo Similaire
Énergie 0.0002350 kWh 0.0000037 kWh 98.4%
CO₂ 0.1346 g 0.0021 g 98.4%

Équivalence concrète (Green IT) :

Pour un centre de données traitant 10 To de logs par jour :

  • Économie CO₂ : 4,835 kg CO₂/an (équivalent à 43,170 km en voiture selon ADEME)
  • Équivalence française : 11.5 fois l'empreinte numérique annuelle d'un Français
  • Réduction I/O : 262 millions d'appels système évités quotidiennement
  • Durée de vie SSD : Prolongée de 40% (moins d'écritures fragmentées)

#concurrence #cpu #gil #calcul-parallele
52.5%
Gain CO₂
2.5×
Plus rapide
⭐⭐⭐⭐

Principe

Principe

Utiliser @lru_cache pour mémoriser les résultats de fonctions pures coûteuses, avec limite de taille pour éviter la fuite mémoire.

Anti-pattern (Recalcule à chaque fois) : 

# Recalcul systématique - exponentiel O(2^n)

@lru_cache(maxsize=128)
def fibonacci_rapide(n):  # O(n) unique, puis O(1) lookup
    if n < 2:
        return n
    return fibonacci_rapide(n - 1) + fibonacci_rapide(n - 2)

Explication

  • Sans cache : Complexité exponentielle O(2^n), redondance massive des calculs
  • Avec cache : O(n) unique, puis O(1) lookup — les résultats sont stockés

Principe

Cette règle regroupe les micro-optimisations essentielles et les bonnes pratiques de style (PEP8) pour un code Python propre, efficace et maintenable. L'application systématique de ces pratiques peut réduire la consommation énergétique globale de 10-20% sur une codebase complète.

10.1 Variables locales vs globales

Principe : Les accès aux variables locales sont plus rapides que les accès aux variables globales ou aux attributs d'objets, car Python utilise le dictionnaire des variables locales (optimisé) plutôt que la recherche dans le dictionnaire global.

# Mauvais : accès global répété dans une boucle
import math

def process_data_slow(data):
    result = []
    for item in data:
        result.append(math.sqrt(item))  # Recherche math.sqrt à chaque itération
    return result
# Bon : variable locale pour éviter les recherches répétées
import math

def process_data_fast(data):
    sqrt = math.sqrt  # Capture en variable locale
    result = []
    for item in data:
        result.append(sqrt(item))  # Accès direct O(1)
    return result

# Encore mieux : list comprehension (plus rapide et pythonique)
def process_data_best(data):
    sqrt = math.sqrt
    return [sqrt(item) for item in data]

Impact : Jusqu'à 15-20% de gain de performance dans les boucles serrées. Sur une codebase complète, cela peut représenter une économie significative d'énergie.

10.2 Imports au niveau module

Principe : Importer les modules au niveau global (début du fichier) évite les coûts répétés d'importation dans les fonctions appelées fréquemment. Le module n'est chargé qu'une seule fois et mis en cache dans sys.modules.

# Mauvais : import répété à chaque appel de fonction
def parse_json_slow(data):
    import json  # Coût d'import à chaque appel!
    return json.loads(data)

# Appelé 1 million de fois = 1 million d'imports inutiles
# Bon : import au niveau module (une seule fois)
import json  # Au début du fichier - exécuté une seule fois

def parse_json_fast(data):
    return json.loads(data)  # Réutilise le module déjà chargé

10.3 Gestion des ressources avec with

Principe : Le context manager with garantit la libération immédiate des ressources (fichiers, connexions, verrous) même en cas d'exception, évitant les fuites de ressources qui peuvent dégrader les performances et la stabilité.

# Mauvaise pratique : ne garantit pas la fermeture
f = open('data.txt', 'r')
data = f.read()
# Si exception ici, le fichier reste ouvert (fuite de ressources)
f.close()  # Peut ne jamais être exécuté
# Bon : garantit la fermeture même en cas d'erreur
with open('data.txt', 'r') as f:
    data = f.read()
# Fichier automatiquement fermé, même si une exception est levée

# Fonctionne aussi avec les connexions réseau, les verrous, etc.
from threading import Lock
lock = Lock()

with lock:
    # Section critique - le verrou est automatiquement libéré
    modify_shared_resource()

10.4 List comprehensions vs boucles for

Principe : Les list comprehensions sont optimisées en C et sont généralement 2-3× plus rapides qu'une boucle for équivalente. Elles sont également plus lisibles et pythoniques.

# Moins efficace et moins lisible
result = []
for x in data:
    if x > 0:
        result.append(x * 2)
# Plus rapide et plus pythonique
result = [x * 2 for x in data if x > 0]

# Pour les très grandes collections, utiliser un générateur
result = (x * 2 for x in data if x > 0)  # O(1) mémoire

10.5 F-strings vs .format() vs %

10.7 PEP8 - Style Guide for Python Code

PEP8 est le guide de style officiel pour Python. Un code conforme à PEP8 est :

  • Plus lisible : Facilite la maintenance et réduit les bugs
  • Plus cohérent : Uniformité entre les projets Python
  • Plus maintenable : Réduit le temps de compréhension du code
# Règles PEP8 essentielles

# 1. Indentation : 4 espaces (pas de tabulations)
def ma_fonction():
    pass  # 4 espaces

# 2. Longueur de ligne : maximum 79 caractères (88 avec Black)

x=a+b*c        # Incorrect

# 5. Noms des variables : snake_case pour les fonctions/variables
def calculer_moyenne(liste_valeurs):
    pass

class CalculateurStatistique:
    pass

# 7. Constantes : UPPER_CASE

# Au lieu de :
for i in range(len(items)):
    process(i, items[i])

# Préférer :
for i, item in enumerate(items):
    process(i, item)

# Utiliser zip() pour itérer sur plusieurs séquences
noms = ["Alice", "Bob", "Charlie"]
ages = [25, 30, 35]

for nom, age in zip(noms, ages):
    print(f"{nom} a {age} ans")

# Utiliser dict.get() avec valeur par défaut
value = my_dict.get("key", "default")  # Évite KeyError

# Utiliser setdefault() pour les dictionnaires de listes
from collections import defaultdict
# Au lieu de vérifier si la clé existe
groups = defaultdict(list)
for key, value in items:
    groups[key].append(value)  # Crée la liste automatiquement
